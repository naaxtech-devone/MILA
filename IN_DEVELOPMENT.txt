MILA — IN DEVELOPMENT FEATURES
Last updated: 2026-07-13

Purpose
-------
This file lists functionality intentionally disabled or marked as in
development for the current deployment. It was produced by a full-repository
audit against the actual code (not the README) — only confirmed unfinished
functionality is listed here. Completed functionality is not included.

Status definitions
------------------
PARTIAL      Some functionality works, but specific actions remain unavailable.
DEVELOPMENT  The feature is visible but not complete.
DISABLED     The feature is intentionally unavailable in this deployment.

FEATURE: credit-purchases
STATUS: DEVELOPMENT

User-facing area:
- "Studio Energy Depleted" credit-pack dialog (opened only when a credit
  paywall would trigger — i.e. a genuine insufficient-credits failure from
  the Studio Lens or dashboard generation). The header credit counter no
  longer opens this dialog; it now navigates to the /pricing membership
  page (see the membership-passes update below).

Routes:
- /dashboard (and any other authenticated page via the shared app shell)

Source files:
- src/components/dashboard/upgrade-slots-dialog.tsx
- src/constants/app.ts (CREDIT_PACKS data, unchanged — still used for preview)

Deployment behavior:
- The three credit-pack buttons are rendered with the native `disabled`
  attribute and reduced opacity — they no longer look or behave as
  clickable.
- An "In development" badge is shown in the dialog header.
- A DevelopmentNotice explains that purchasing isn't available yet and
  that existing daily credits still work.
- No toast-as-fake-success and no network/payment call is triggered by
  these buttons (previously they only showed a "billing comes online
  soon" toast — that fake-success-shaped interaction was removed).

Reason:
- No payment provider (Stripe or otherwise) is integrated anywhere in the
  repository. The `purchases` table exists in Postgres but has no client
  writer and no webhook — by design, its RLS policy only grants SELECT to
  authenticated users, so a purchase row can only ever be created by a
  trusted server process that does not yet exist.

Remaining work:
- Choose and configure a payment provider.
- Add server-side purchase verification (webhook-driven, not
  client-reported).
- Connect verified purchases to `user_entitlements` updates.
- Add failure and refund handling.

FEATURE: credit-enforcement
STATUS: DEVELOPMENT

User-facing area:
- None directly — this is a backend behavior. The credit counter shown in
  the app header ("N AI credits") reads a real `user_entitlements.
  ai_credits` value, but that value never decreases because consumption
  is not yet wired up.

Routes:
- All authenticated routes (via the shared app shell credit counter)

Source files:
- src/lib/credits.server.ts (consumeAiCredit — always returns 999,
  never deducts)
- src/lib/credits.ts (isInsufficientCreditsError — always returns false,
  so the paywall branch that opens the credit-purchases dialog is
  currently unreachable)

Deployment behavior:
- AI-powered features (Generate Look, Lens outfit analysis, Concierge
  chat, Dupe Hunter) remain fully enabled and usable — per the
  deployment brief, credit enforcement being incomplete must not block
  otherwise-working AI functionality.
- The credit counter continues to display a real (not hardcoded) number
  from the database; it just does not yet reflect real usage.
- Source comments were added so this is not mistaken for an oversight.

Reason:
- The premium/credit gate was intentionally stubbed out in prior work
  ("Re-enable premium gate before production launch") pending a real
  purchase flow (see credit-purchases above) — enforcing a limit with no
  way to buy more credits would be a dead end for users.

Remaining work:
- Re-enable real credit deduction in consumeAiCredit.
- Re-enable the insufficient-credits check once credit-purchases ships.
- Add a credits ledger table if historical "credits issued" reporting is
  needed (the admin dashboard currently shows a running balance, not a
  historical total — see src/lib/admin.functions.ts:370-371).

COMPLETED 2026-07-13: concierge-chat (removed from this list)

Mila's Styling Studio is now a real AI chat and the Concierge nav
entries (desktop + mobile) are re-enabled. Both modes call the real
model through one server function (src/lib/concierge-chat.functions.ts):
general styling questions (no image) grounded in the server-loaded
style profile, and look-anchored questions where the server resolves
the image and details from the user's own `outfits` row (ownership
enforced). Entry points: nav (general), History detail, a just-saved
dashboard look, and the Lens save toast. The old `fixOutfitChat`
image-only path was deleted.

Still intentionally out of scope for concierge-chat:
- Conversation persistence across sessions (state is in-session only;
  no concierge tables exist yet — do not claim otherwise).
- The rate limiter (src/lib/rate-limit.server.ts) is per-instance
  in-memory; consolidate onto a durable store with the broader
  rate-limiting task.
- Credit enforcement remains stubbed (see credit-enforcement below) —
  Concierge calls consumeAiCredit by convention but nothing deducts.

FEATURE: moderator-role
STATUS: DEVELOPMENT

User-facing area:
- Admin → Members page (development notice shown above the members
  table).

Routes:
- /admin/members

Source files:
- src/routes/_authenticated/admin/members.tsx

Deployment behavior:
- A DevelopmentNotice is shown on the Members page clarifying that only
  the Steward (admin) role can be granted in this release.
- No UI change was needed to "disable" a moderator-assignment control,
  because none exists — the notice exists to set correct expectations,
  not to hide a broken control.
- All admin functionality that IS complete (Steward grant/revoke, member
  suspend/reinstate, member create/edit, moderation queue, support inbox)
  remains fully working and was not touched.

Reason:
- The `moderator` value exists in the Postgres `app_role` enum, but no
  application code reads, assigns, or authorizes against it — every
  admin-authorization check (`assertAdmin`, the `has_role` RPC call, the
  admin route guards) is hardcoded to check for `'admin'` only. It is a
  dead enum value today.

Remaining work:
- Design moderator-specific permissions (a defined subset of admin
  capabilities).
- Add a role-assignment control distinct from the Steward toggle.
- Add moderator-aware authorization checks alongside the existing
  admin-only checks, without weakening them.
